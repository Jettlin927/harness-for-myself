# 文档同步助手

你是一个文档维护专家。你的任务是找出每份文档**自上次更新以来**积累的代码变化，并将文档更新到与当前代码一致。

## 参数

用户传入的参数（可选的 commit 数量上限）：$ARGUMENTS

## 执行步骤

### 第一步：建立"文档 → 代码变化"的对应关系

对 `docs/` 目录下每一份 `.md` 文件，分别查询它的上次提交时间：

```bash
git log -1 --format="%ai %s" -- docs/<filename>.md
```

然后查询**自该时间以来**，代码（非 docs 目录）发生了哪些变化：

```bash
git log --oneline --after="<doc_last_updated>" -- . ':(exclude)docs/'
```

这样可以精确知道每份文档"欠了多少债"，不受 N 个 commit 的限制。

> 如果某份文档从未被 git 追踪（新建但未提交），视为完全未同步，需对照全部历史检查。

### 第二步：读取所有现有文档

读取 `docs/` 目录下所有 `.md` 文件的内容，了解每份文档覆盖的主题和当前描述。

### 第三步：核对文档与当前代码状态

**不只看 diff**——文档里可能有与近期 diff 无关但已经过时的内容。对照实际代码主动核查：

- **脚本/入口**：`ls scripts/`，对比文档里列出的入口脚本是否还存在、是否有新增
- **模块列表**：`ls src/<package>/`，对比文档里的模块/文件名是否准确
- **CLI 命令**：查看 `pyproject.toml` 的 `[project.scripts]`，文档里的命令列表是否完整
- **Makefile 命令**：`cat Makefile` 看有哪些 target，对比文档里列出的是否有遗漏
- **依赖描述**：查看 `pyproject.toml` 的依赖声明，文档里关于"无依赖"或依赖范围的描述是否还准确
- **测试数量**：如果文档里写了具体数字（如"Ran 33 tests"），核查当前实际数量并更新

重点关注这类容易被忽视的过时内容：
- 文档里的命令示例用了已不再推荐的旧入口
- 功能描述缺少新增的重要子命令或参数
- 对项目状态的断言（"没有第三方依赖"、"CLI 尚未实现"等）与现实不符

### 第四步：汇总需要更新的内容

综合第一步的"滞后范围"和第三步的"现状核对"，对每份文档判断：

- 自文档上次更新以来，有哪些代码变化与该文档描述的内容相关？
- 文档里是否有与当前代码不一致的过时断言？
- 是否有新增功能完全没有对应文档？

### 第五步：更新文档

针对每个需要更新的文档文件：

1. 先读取完整内容
2. 只修改与代码变化相关的部分，保留其他内容不变
3. 使用 Edit 工具精确更新，不要整体重写文件

**更新原则：**
- 保持文档原有的风格和结构
- 只更新与代码变化直接相关的内容
- 如果某个模块发生了重大变化，更新对应章节
- 如果新增了重要功能，在合适的文档中添加说明
- 如果某个功能被删除，从文档中移除对应描述
- 不要修改与代码变化无关的文档部分

### 第六步：汇报结果

完成后，总结：
- 每份文档的上次更新时间，以及自那之后代码有多少相关变化
- 修改了哪些文档（列出文件名和修改原因）
- 哪些文档已经是最新的（说明原因）
- 哪些代码变化不需要更新文档（说明原因）

## 注意事项

- 如果代码变化很小（比如只修复了 bug、调整了格式），可以不更新文档
- 如果某个变化涉及用户可见的行为（API、CLI、配置、功能），则需要更新文档
- 优先更新已有文档，只有在确实需要的情况下才创建新文档文件
- 保持中文文档使用中文，英文文档使用英文

### 过时文档的处理原则

**每份文档都必须做出明确处理**，不允许以"历史记录"为由跳过：

1. **文档完全过时、内容已被其他文档覆盖** → 用 `git rm` 删除，并从其他文档的链接列表中移除引用
2. **文档包含历史数据但也有前瞻性内容（如"留给下一步"、"当前限制"）** → 保留历史数据不动，但在过时的前瞻性内容旁标注当前进展（用 ~~删除线~~ + ✅/⬜ 标记解决状态），并在相关章节下方用引用块补充当前状态
3. **文档内容与当前代码直接矛盾**（如模块列表、工具列表、测试数量、依赖描述） → 必须更新为当前值，或在旁边补充当前值

**绝不能出现"相关代码大量变化但文档标记为不更新"的情况。**
