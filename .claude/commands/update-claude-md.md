# CLAUDE.md 更新助手

你的任务是更新项目根目录的 `CLAUDE.md`，使每个新 session 都能快速了解项目现状、下一步方向、以及去哪里找细节。

## 设计理念

CLAUDE.md 是**索引层**，不是**内容层**。遵循以下原则：

### 原则一：只放「不看代码就不知道」的信息

- **放：** 项目定位、当前进展阶段、下一步优先级、开发约定
- **不放：** 具体函数签名、代码片段、API 参数——这些直接读源码更准确
- **判断标准：** 如果信息能通过 `grep`/`glob`/读文件在 10 秒内找到，就不写进 CLAUDE.md

### 原则二：用文件引用代替内容复制

- 每个主题指向对应的 `docs/*.md` 或源码路径，而非在 CLAUDE.md 里重复内容
- 好处：避免 CLAUDE.md 与源码/docs 之间的信息不一致
- 格式示例：`详见 docs/evolution-roadmap.md → Phase 2`

### 原则三：进展状态要具体、可验证

- 用 ✅ / 🚧 / ❌ 标记每个阶段/模块的状态
- "进行中"要说明**已完成什么、还差什么**，不能只写"进行中"
- 状态描述要能让下一个 session 立刻判断：是继续当前工作，还是开始新阶段

### 原则四：覆盖「新 session 冷启动」所需的全部维度

一个新 session 需要知道的最小信息集：

1. **项目是什么** — 一句话定位 + 技术栈
2. **现在做到哪了** — 阶段状态表，指向详情文档
3. **下一步做什么** — 按优先级排列，指向计划文档
4. **代码在哪** — 模块结构 + 职责速查（不含实现细节）
5. **文档在哪** — docs 索引表，说明每份文档覆盖什么
6. **怎么开发** — 常用命令、代码风格、测试要求、类型约定

### 原则五：测试和质量要求不能省

- 明确写出测试框架、运行方式、覆盖标准
- 写清楚测试分层（冒烟/单元/回归/评估）
- 写清楚每个模块对应的测试文件
- 这些信息决定了 Claude 在写完代码后如何验证

### 原则六：保持精简，控制长度

- 目标：CLAUDE.md 在 150 行以内
- 用表格代替长段落
- 每个段落不超过 3 行说明
- 如果某个主题需要超过 5 行才能说清楚，说明它应该在 docs/ 里而不是 CLAUDE.md 里

## 执行步骤

### 第一步：收集现状

并行执行以下操作：

1. 读取当前 `CLAUDE.md`（如果存在）
2. 读取 `docs/` 下所有 `.md` 文件，提取各阶段状态
3. `ls src/harness/` 获取最新模块列表
4. `ls tests/` 获取最新测试文件列表
5. 读取 `pyproject.toml` 获取依赖和工具配置
6. 读取 `Makefile` 获取开发命令
7. `git log --oneline -10` 了解最近动态
8. `git status --short` 了解未提交的工作

### 第二步：对照六大原则逐项检查

对 CLAUDE.md 的每个章节，检查：

- [ ] 是否有可以用文件引用替代的内容复制？
- [ ] 进展状态是否具体到「已完成/未完成」的颗粒度？
- [ ] 是否遗漏了新增的模块、测试文件、或开发命令？
- [ ] 测试相关描述是否完整（框架、命令、分层、标准）？
- [ ] 是否超过 200 行？哪些内容可以精简或外移到 docs？

### 第三步：更新 CLAUDE.md

用 Edit 工具精确修改，不要整体重写。重点更新：

1. **进展状态表** — 根据 docs 和代码刷新 ✅/🚧 标记
2. **进行中工作** — 根据 git status 和最近 commits 更新细节
3. **模块结构** — 如有新增/删除模块则同步
4. **测试文件列表** — 如有新增测试则同步
5. **开发命令** — 如 Makefile 有变化则同步

### 第四步：汇报

总结本次更新了什么、为什么更新、以及哪些部分确认无需变化。
